# 1. Целевая архитектура данных через год

Архитектура через год должна быть ориентирована на данные, масштабируема и позволять независимое развитие доменов. Я предлагаю переход от монолитного DWH к Data Mesh-подходу и современному Data Lakehouse.

## Пояснения к архитектуре

# Архитектура платформы данных

## 1. Слой: Data Producers (Источники данных / Бизнес-домены)
Это внешние по отношению к платформе системы, которые потребляются как контейнеры.

| Контейнер | Технологии / Примеры | Описание |
|-----------|----------------------|----------|
| Финтех-сервисы | Golang, Java, Kafka API | Банковские и финансовые приложения. Публикуют данные о транзакциях, счетах, кредитах через API/события. |
| Медицинская ИС Клиник | Power Builder, MS SQL Server App Layer | Легаси-приложение для операционной деятельности клиник. Источник данных о пациентах, расписаниях, инвентаре. |
| ИИ-Сервисы | Python, ML-фреймворки | Сервисы машинного обучения. Публикуют результаты анализов, прогнозные модели и производные insights. |
| Новые бизнесы | (Различные, напр., Java, .NET) | Фармацевтические компании и производители электроники. Интегрируются как новые независимые источники. |

## 2. Слой: Data Infrastructure Platform (Платформа данных)
Это ядро новой архитектуры, предоставляющее capabilities как сервис.

| Контейнер | Технологии / Примеры | Описание |
|-----------|----------------------|----------|
| Data Ingestion API / Gateway | REST API, Kafka Connect, Debezium | Единая точка входа для приема данных от всех производителей. Поддерживает как потоковый (Kafka), так и пакетный (API) режимы. |
| Orchestration Service | Apache Airflow, Prefect, Dagster | Сервис для оркестрации ETL/ELT-пайплайнов. Позволяет доменам автоматизировать процессы преобразования своих данных в Data Products. |
| Processing Engine | Apache Spark, Databricks Runtime, Flink | Высопроизводительный движок для распределенной обработки больших объемов данных. Используется оркестратором для выполнения задач трансформации. |
| Unified Data Catalog | Databricks Unity Catalog, Open Metadata, AWS Glue Catalog | "Мозг" платформы. Регистрирует все активы данных (таблицы, файлы), их метаданные, lineage и политики доступа. Предоставляет API для обнаружения данных. |
| Object Storage (Data Lakehouse) | AWS S3, Azure ADLS Gen2, Delta Lake, Iceberg | Центральное, дешевое и неограниченно масштабируемое хранилище для всех данных в открытых форматах. Заменяет собой DWH. Физическое место хранения Data Products. |
| Data Access & Security Manager | RBAC-системы, Apache Ranger | Централизованный сервис управления доступом. Интегрирован с каталогом и SQL-движками для enforcement политик на уровне строк и столбцов. |

## 3. Слой: Data Products (Продукты Данных)
Это логические артефакты, а не физические контейнеры. Они материализуются в Object Storage и регистрируются в Unified Catalog.

| Логический Data Product | Владелец (Домен) | Описание | Местоположение (в Object Storage) |
|-------------------------|------------------|----------|-----------------------------------|
| Finance Mart | Финтех-сервисы | Очищенные, готовые к использованию данные по клиентам, счетам и транзакциям. | `s3://data-products/finance/` |
| Hospital Operations Mart | Клиники | Агрегированные данные о работе клиник: расписания, использование инвентаря, кадровые данные. | `s3://data-products/hospital/` |
| AI Insights Mart | ИИ-Сервисы | Результаты прогнозов, аномалии и другие производные данные, готовые для аналитики. | `s3://data-products/ai/` |

## 4. Слой: Data Consumption (Потребление данных)

| Контейнер | Технологии / Примеры | Описание |
|-----------|----------------------|----------|
| SQL Query Engine | Trino, Presto, Serverless Spark, Databricks SQL | Высокоуровневый сервис для выполнения SQL-запросов к данным в Object Storage. Предоставляет интерфейс, привычный для аналитиков. |
| BI & Visualization Portal | Power BI Service, Tableau Online, Metabase | ВИтрина данных самообслуживания. Основной инструмент для бизнес-пользователей. Подключается к SQL Engine и каталогу для построения отчетов. |
| Legacy BI & Reports | Power Builder Reports, старые SSRS reports | Легаси-отчеты, которые на переходный период могут продолжать работать со старым DWH через специальные коннекторы. |
| Data Science Workspace | JupyterHub, Databricks Notebooks | Среда для data scientists для исследования данных, построения и эксплуатации ML-моделей на основе данных из Lakehouse. |
| Legacy DWH (MS SQL Server 2008) | MS SQL Server 2008 | Изолированный легаси-контейнер. Используется только для обратной совместимости. Направление новых данных в него прекращено. Цель — полный вывод из эксплуатации. |

## 5. Слой: Legacy & Integration (Легаси и Интеграция)

| Контейнер | Технологии | Описание |
|-----------|------------|----------|
| Legacy ESB | Apache Camel | Существующая шина данных. Изолирована. Обслуживает только старые, не мигрировавшие интеграции. Постепенно замещается современными паттернами. |
| Legacy DWH DB | MS SQL Server 2008 | База данных старого DWH. Рассматривается как еще один источник данных для миграции на новую платформу. |

# 2. Проблемные места (сформулированы для бизнеса и инженеров)

| Проблема | Описание для бизнеса | Техническая суть |
|----------|----------------------|------------------|
| **Устаревшая и монолитная DWH** | Наш "мозговой центр" данных (DWH) стар, медленен и не справляется с ростом. Сложные отчеты готовятся часами, что мешает оперативно принимать решения. Любое изменение требует много времени и сил разработчиков. | SQL Server 2008 устарел, не поддерживает современные форматы данных и горизонтальное масштабирование. Высокая связность бизнес-логики делает систему хрупкой. |
| **Смешение ответственности** | Все данные свалены в одну кучу. Непонятно, кто отвечает за актуальность финансовых данных, а кто — за данные инвентаризации. Это приводит к ошибкам и долгим согласованиям. | Отсутствует принцип domain-oriented ownership. DWH является единой точкой отказа и узким горлом в разработке. |
| **Низкая скорость разработки (Time-to-Market)** | Чтобы добавить данные от новой аптеки или банка, нужно несколько месяцев работы и рисковать стабильностью всех текущих отчетов. | Любая интеграция требует написания сложной ETL-логики внутри DWH, что замедляет разработку и повышает риски. |
| **Угроза безопасности и compliance** | Медицинские и финансовые данные требуют особой защиты по закону. В текущей системе сложно гибко управлять доступом, так как все данные хранятся вместе. | Высокий риск несанкционированного доступа из-за монолитной модели хранения. Сложность выполнения требований GDPR, HIPAA и др. |
| **Дорогая и неэффективная поддержка** | Мы тратим значительные ресурсы не на развитие аналитики, а на поддержку работы старой системы и "костылей" вокруг нее. | Высокие эксплуатационные расходы на поддержку legacy-инфраструктуры (SQL Server 2008, Power Builder). |
| **Сложность самообслуживания** | Существующие BI-инструменты не дают реальной возможности самостоятельной работы с данными, так как данные неструктурированы и недоверены. | Отсутствие каталога данных, единых метрик и доверенных продуктов данных приводит к тому, что пользователи не могут сами строить отчеты. |


# 3. Приоритизация проблем (Метод MoSCoW)

### MUST HAVE (Обязательно к реализации в первую очередь)

- **Создание современного масштабируемого хранилища данных (Data Lakehouse)**
  Это фундамент для всех дальнейших улучшений. Позволит drastically увеличить скорость выполнения запросов и снизить стоимость хранения.

- **Внедрение принципов Data Mesh и организация данных по доменам**
  Это решит проблемы с ответственностью, ускорит интеграцию новых бизнесов и повысит качество данных.

- **Развертывание Единого Каталога Данных (Unified Catalog)**
  Без этого невозможна "витрина самообслуживания". Это основа для обнаружения и понимания данных бизнес-пользователями.

### SHOULD HAVE (Желательно реализовать вслед за первым этапом)

- **Разработка и миграция на портал самообслуживания**
  На основе новых Data Products и современного BI-инструмента (Power BI Premium/Fabric).

- **Построение современного слоя оркестрации и ETL/ELT**
  Например, на Apache Airflow/Apache Spark для автоматизации процессов обработки данных.

- **Начать миграцию нагрузок и отчетов со старого DWH на новую платформу**

### COULD HAVE (Можно реализовать при наличии ресурсов)

- **Внедрение событийной архитектуры**
  Например, с помощью Apache Kafka для обработки данных в реальном времени от финтех-сервисов и медицинского оборудования.

- **Расширение платформы для MLops**
  Чтобы упростить работу ИИ-сервисов с данными в Lakehouse.

### WON'T HAVE (Не в рамках данной инициативы, но важно помнить)

- **Мгновенный полный отказ от всех легаси-систем**
  Это будет постепенный процесс миграции в течение всего года. На первом этапе некоторые системы (например, шина Camel) могут работать в режиме совместимости.
